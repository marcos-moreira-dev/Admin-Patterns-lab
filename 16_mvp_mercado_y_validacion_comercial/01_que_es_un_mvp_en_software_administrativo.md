# Qué es un MVP en software administrativo

## Idea central

En software administrativo, un MVP no es una aplicación "pequeña" por tamaño ni una maqueta fea a la que le faltan cosas. Es la **mínima versión operativa** que resuelve un flujo importante del negocio, reduce fricción real y permite validar si vale la pena seguir invirtiendo.

En este contexto, un MVP debe poder usarse. No basta con que se vea bien o con que tenga una arquitectura elegante. Debe ayudar a que el negocio haga mejor una tarea concreta.

## Qué sí es un MVP

Un MVP administrativo suele incluir:

- un flujo principal resuelto de inicio a fin
- uno o dos actores claros
- estados mínimos para que el proceso tenga control
- registro confiable de información importante
- búsqueda o consulta básica
- algún comprobante, resumen o reporte simple si aporta valor

Ejemplos de flujo principal:

- en una óptica: registrar pedido, dejar abono, actualizar estado y entregar
- en reparación de celulares: recepción, diagnóstico, aprobación y entrega
- en un consultorio: cita, atención, registro y cobro
- en una tienda pequeña: venta, actualización de stock y comprobante simple

## Qué no es un MVP

No es:

- un sistema completo con todos los sueños del cliente
- una app vacía con muchas pantallas decorativas
- una arquitectura compleja montada "por si acaso"
- una demo desconectada de la operación real
- una versión tan recortada que no se puede cobrar ni defender

## Diferencia entre MVP, demo y V1

### Demo
Sirve para enseñar una idea, probar interfaz o comunicar una visión. Puede no tener valor operativo real.

### MVP
Ya resuelve un núcleo del negocio. Se puede usar en una operación pequeña o piloto. Sirve para validar y para cobrar una primera fase.

### V1
Es una versión más seria y estable. Ya incorpora aprendizajes del uso real, mejor recorte funcional, mejores validaciones y un poco más de formalidad.

## Qué debe lograr un buen MVP

Un buen MVP administrativo debe lograr al menos una de estas cosas:

- ahorrar tiempo
- reducir errores manuales
- dar trazabilidad mínima
- reemplazar una parte del caos operativo
- evitar depender de mensajes, papeles o Excel para una tarea crítica
- dar información básica para decidir mejor

## Qué preguntas ayudan a definirlo

- ¿Cuál es el flujo más repetido o más doloroso del negocio?
- ¿Qué error ocurre con más frecuencia hoy?
- ¿Qué parte del trabajo sí o sí debería quedar registrada?
- ¿Qué cosa, si queda ordenada, ya mejora bastante la operación?
- ¿Qué sería suficientemente útil como para que el cliente diga "esto ya me sirve"?

## Criterio práctico

El MVP no debe intentar cubrir toda la empresa. Debe cubrir **el núcleo más defendible**. La clave no es meter muchas funciones, sino elegir bien cuál problema sí vale la pena resolver primero.

## Señal de que el MVP está bien recortado

Si puedes explicarlo así, vas bien:

> Esta primera versión no busca cubrir todo el negocio. Busca resolver de forma sólida el flujo más importante para empezar a ordenar la operación y sentar una base usable.
