# Cómo aprovechar esta biblioteca

Esta biblioteca no está pensada solo para leerla como si fuera un manual académico. Está pensada para acompañar decisiones reales de trabajo.

## Enfoque educativo

Desde el punto de vista educativo, conviene usarla para construir una forma de pensar más clara.

La lectura correcta no es memorizar todos los términos ni recorrer todas las carpetas de principio a fin. Lo más útil es entender la cadena principal:

1. negocio;
2. levantamiento;
3. modelado;
4. datos;
5. backend;
6. interacción;
7. despliegue;
8. mantenimiento;
9. evolución.

Si un bloque todavía se siente muy abstracto, conviene volver al caso real y releer ese bloque con una situación concreta en mente.

## Enfoque operativo

Desde el punto de vista operativo, la biblioteca sirve mejor cuando se convierte en herramienta de trabajo.

Eso significa:

- usar los documentos como guía para reuniones;
- tomar preguntas y convertirlas en checklist;
- usar las plantillas para documentar casos reales;
- apoyarse en el glosario cuando aparezcan términos borrosos;
- volver a los bloques de MVP, cliente y evolución cuando haya que cortar alcance o justificar una nueva fase.

## Qué hacer antes de un proyecto real

Antes de hablar con un cliente real, la secuencia mínima recomendada es:

1. revisar `00_norte_y_criterio/00_mapa_general.md`;
2. repasar `01_levantamiento_de_informacion/`;
3. revisar `06_relacion_con_cliente/`;
4. mirar `16_mvp_mercado_y_validacion_comercial/`;
5. tener abiertas `98_glosario/` y `99_plantillas/`.

## Qué hacer durante un proyecto real

Durante un proyecto, conviene tratar la biblioteca como apoyo contextual.

Eso implica:

- consultar solo el bloque que toca en ese momento;
- no sobredocumentar antes de tiempo;
- comparar lo que el cliente dice con los bloques de dominio, flujo, módulos y reglas;
- volver a la biblioteca cuando aparezcan dudas de criterio, no solo dudas de definición.

## Qué hacer después de cada proyecto

Después de cada proyecto o fase, conviene usar la biblioteca como herramienta de cierre.

Preguntas útiles:

- ¿qué parte del proceso quedó más clara?
- ¿qué parte todavía se improvisó demasiado?
- ¿qué sección del repositorio debería releerse antes del siguiente cliente?
- ¿qué glosario faltó dominar mejor?
- ¿qué plantilla habría ahorrado tiempo si se usaba antes?

## Criterio general

La biblioteca sirve mejor cuando ayuda a tomar decisiones más sobrias, no cuando se convierte en otra fuente de ansiedad técnica.

La meta no es sonar más sofisticado. La meta es trabajar con más claridad, cortar mejor el alcance, construir versiones defendibles y evolucionar el software sin perder tiempo en complejidad que todavía no hace falta.
