# Licencias de software, fuentes, imágenes y assets

## Idea base
No solo el código tiene licencias. Casi cualquier activo externo puede imponer condiciones.

## Qué entra aquí
- librerías y frameworks
- tipografías
- íconos
- imágenes y fotografías
- sonidos y música
- plantillas
- fragmentos de código copiados o adaptados
- modelos, datasets o archivos de configuración de terceros

## Qué debes registrar mínimamente
- nombre del activo
- autor o fuente
- tipo de licencia
- si permite uso comercial
- si exige atribución
- si permite modificación
- si prohíbe redistribución
- dónde se usa en el proyecto

## Tipos de situación típicos
### Software open source
Debes revisar obligaciones de distribución, avisos, copyleft y compatibilidad con tu forma de entregar el producto.

### Tipografías
No asumas que toda fuente se puede usar y redistribuir dentro del instalador o del frontend.

### Imágenes e íconos
Conviene usar familias y repositorios con licencia clara. No copiar de buscadores sin verificar nada.

### Assets generados por terceros
Aunque los uses como apoyo visual, igual conviene dejar rastro de origen y condiciones.

## Qué hacer en la práctica
- mantener una lista de assets y dependencias relevantes
- guardar enlaces o referencias de licencia
- evitar meter assets dudosos en repos públicos o productos distribuidos
- revisar si el cliente espera propiedad total o solo uso del software

## Señales de alerta
- 'lo saqué de internet y ya'
- tipografías incluidas sin revisar redistribución
- imágenes de marcas o productos sin criterio
- paquetes con licencias incompatibles con tu forma de distribución

## Regla práctica
Si un activo no es tuyo y va a entrar al producto, debe quedar identificado y justificado.
